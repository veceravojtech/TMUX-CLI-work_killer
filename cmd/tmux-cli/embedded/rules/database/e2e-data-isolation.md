# E2E data isolation (pack: database — HAS_DATABASE)

Binding planning convention, loaded when the project has a database.
Extracted verbatim from the planner's `<conventions>` block.

<rule critical="true" id="E2E-DATA-ISOLATION-CONV" condition="HAS_DATABASE">E2E DATA ISOLATION — BINDING ON E2E-GENERATING STEPS (3.18, 3.19, 3.29). Fixture rows loaded by ensure-stack are READ-ONLY reference data for E2E specs — specs MUST NOT update, delete, or otherwise mutate seeded fixture rows. Each E2E spec creates its OWN mutable test data through the application's UI or API using unique keys (timestamp or UUID suffix, e.g. `Product-{uuid}`) so entities are identifiable and non-colliding across specs. Count and list assertions MUST filter to the spec's own created data (e.g. search by the unique key or filter by a known attribute) — never assert raw totals against the full table, because fixture rows and other specs' data coexist. Re-running doctrine:fixtures:load mid-suite is FORBIDDEN — the load is a purge-and-reload that destroys all spec-created data; the reset unit is the whole suite (one load via ensure-stack before the first spec). dama/doctrine-test-bundle transaction rollback does NOT apply to E2E tests — the Playwright/HTTP test runner and the application server use separate DB connections across process boundaries; do NOT prescribe it for E2E isolation.</rule>
